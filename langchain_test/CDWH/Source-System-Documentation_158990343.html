MSSQL-Backupserver

Several MSSQL Source-System-Databases are made available towards CDWH, by restoring backups of Live-Databases to one server (MSSQL-Backupserver). The following databases are restored every night, using a fixed schedule that is maintained and defined by the ICT-Database-Team:

  • emedistaH (ZAMD)
  • CATO_Prod (CATO)
  • MOSAIQ (MOSAIQ)
  • MEONAP (Meona)
  • USB_Basel_Archive (PDMS Metavision)
  • Copra6Live (COPRA)

The names of the databases just partially correspond to the names of the source systems (i.e. USB_Basel_Archive is the database of PDMS Metavision). The schedule may be changed by the ICT-Database-Team by any time, what might impact the LZ schedules in SAP Data Services. Every LZ-Job that depends on such a restored database from the MSSQL-Backupserver, performs a check to ensure that the database is current and fails, if this is not the case. This can happen, when the ICT-Database-Team changes the schedules of the restores, without notifying the CDWH-Team.

To connect to this server, please refer to the entry "MSSQL-Backupserver" in KeePass. To check the available databases and the restore timestamps of databases, that have been restored in the last 24 hours, you can use the following SQL-statement:

SELECT
[rs].[destination_database_name],
[rs].[restore_date],
[bs].[backup_start_date],
[bs].[backup_finish_date]
FROM msdb..restorehistory rs
INNER JOIN msdb..backupset bs ON [rs].[backup_set_id] = [bs].[backup_set_id]
WHERE [rs].[restore_date] >= GETDATE()-1
ORDER BY [rs].[restore_date] ASC

Oracle Databases

  • Connections to Oracle Databases are maintained via TNS-Configuration-File
  • For a proper installation of the Oracle Drivers and the current TNS-Configuration-File, please refer to the following article: Oracle install
  • For the daily loads, we usually connect to the standby-instances of the production databases
  • These are read-only instances, that are frequently updated by applying the redo-logs from the production live-database
  • Standby-instances are available for all Oracle-Source-Databases, except for RIS
  • For some source systems we have credentials in KeePass for both, OLTP- and Standby-Instances
  • Please pay attention when directly connecting to an OLTP-instance, and limit your queries accordingly, in order to keep the impact on the live-system as low as possible!
  • Always use the standby-nodes if possible!

Data Services LZ-Schedules

  • The LZ schedules are maintained manually through the SAP Data Services Management Console
  • Unfortunately, schedules cannot be modified. In order to change the time of a LZ-Load, a new schedule has to be created and the old one must be de-activated/deleted
  • There is limited capacity for loading in parallel, if too many loads are started at the same time, loads can fail
  • Therefore, changing- or adding schedules is difficult and the approach is "try and error"
  • We are already touching the limits and moving schedules in the production environment is a critical task
  • Parallelity is already very high between midnight and the start time of the CDWH-Load (3:40am)
  • If we attach new source systems to the CDWH, we probably have to upgrade the DS-Server or enlarge the time windows for LZ-Loads by moving the schedule of the CDWH-Job

ISOP-Microbatch

This is a requirement from Data-to-Business-Team (DtB-Team):

  • There is a special SAP Data Services repository on PROD: DSREPO_PROD_RT (Real-Time)
  • In this repository, there is only one job running every 2 minutes: BJ_LZ_ISOP_MICROBATCH
  • The job is paused between 3:20am and 4:20am every night, in order to perform the full load of ISOP within this time window
  • The Job inserts & updates the records of table DATO_OP from ISOP

LZ-Schedules DEV

This is a requirement from Data-to-Business-Team (DtB-Team):

  • There is a special SAP Data Services repository on DEV: DSREPO_DEV_DTB
  • There is one daily schedule for BJ_LZ_ISOP: 4:30am
  • There are two daily schedules for BJ_LZ_SERVMAN: 3am & 7am

LZ-Schedules KONS

  • For every schedule on PROD, there is a similar schedule on KONS, with start time +3 hours
    • Exception1: SNOMED and LOINC catalogues are scheduled 40 minutes later than PROD
    • Exception2: There is no schedule for the 2nd SERVMAN-load (7am on PROD)
  • These schedules are usually activated for one week every month, only for CDWH-release testing
