Basic information

  • Check in / check out / update from repository always copy the model between the local workspace (on your VD) and the repository database.
  • Check-in / check-out / update from repository always work on the level of a document (in our case the *.pdm file, i.e. the complete physical data model). In case of differences on some model items between the local workspace and the repository you will be prompted for the appropriate merge action for each item
  • Check out dependencies: applies only for cross-model lineages, in our case irrelevant
  • Check out vs. Update from repository:
    • Check out can only applied to the model in the repository. Check out always creates a new model in your local workspace. If you check out a model twice you will have two models with the same name in your local workspace. You would need to merge these models manually later. Therefore, it is recommended to use the check out command only if the model is not yet in your workspace. Otherwise you can use the Update from repository

    • Update from repository can only applied to a model in your local workspace

  • Check in: when you check in a model into the repository, you have the option to “check out” the model from repository after the check in has completed. This “check out” is actually an update from repository (see notes above), means it does not create a new model in your workspace:

  • .

I recommend to leave this option set. With the check in in you merge your local changes into the repository. Afterwards you would get any other changes that other users made in the meanwhile from the repository

Freezes and Versioning

When you check in a model you have the option to freeze it. That means that the model in the repository will be frozen. Future check ins/changes will generate a new version number. You have the option to freeze the model in the repository before and after the check in:

These two alternatives have an impact on how versions are generated in the repository:

If you always freeze with every check in you will always generate a version number, regardless whether you freeze before or after check in. Because freeze after option is set as default in the check in dialog, I recommend to always use Freeze after check in and never Freeze before check in.

The version number in the model will be applied to all affected items in the object hierarchy. For instance in this test model the column TEST_DETAIL.DETAIL_DESTCRIPTION had been changed 5 times, whereas the other columns have not been changed. Locks

You can a lock a document. This will prevent other users from checking in the model and applying changes. Because of the granular merging mechanism there might not be the need for locking/unlocking documents.

Checking in a model into the repository for the first time

The user who checks in the model for the first time is the owner of the model and has full permission. I particular he can unlock locks from other user. When checking in a model which is locked by another user, the owner just receives a warning message whether the model should be unlocked. This way the owner actually overwrites the intention of the other user. Therefore I recommend that only the ADMIN user should check in a model the first time. Afterwards the permissions for the other users on this model should be set like this:

Then going forward only personal logins should be used to change the model in the repository.

Retrieving a model into the repository for the first time

Retrieve the model for the first time from the repository to your local workspace: check out
Attention: if the check out to file option is checked the model will be saved to the file path. The path might not exist on your VD. In this case you should uncheck the box.

Recommended workflow for changing a model

  1. Get the latest version from the repository:
    select the model in your local workspace and apply Update from repository. The Check out to File check box is activated only if the current model in your workplace has already been saved to a file. Use the default settings in the dialog:
  2. Apply the model changes in your local workplace
  3. Check in your local model into the repository:

    Use the default settings in the dialog. You may want to change the comments filed to annotate your changes:



    In any way the merge changes dialog appears, even if there are no differences between your local model and the model in the repository (see section Merge changes dialog).

Merge changes dialog

The check in process is per default an interactive process. This mode can be changed to Batch by clicking the Options button on Check In Document dialog:

I recommend to use the interactive mode at least at the beginning until you are more familiar with this process.

In interactive mode the Merge Changes dialog opens. You can then explore the object hierarchy to navigate to those items that differ. You may also use the navigation buttons on the top to directly jump to differences. The lower panel shows the details of the differences. You can deselect the item if you don’t want this item change to be applied to the repository:

Please note that the Merge Changes dialog usually activates all changes per default. Nevertheless it may happen that some changes are deselected. This happens when there is a newer change on either side (local or in the repository) which would be overwritten by an older setting on the other side. For instance, the Merge Changes doalog below shows the proposed changes after an Update from repository action:

  • there had beeen recent changes on the CAS_BK column in the repository (probably by another user). PowerDesigner proposes to apply this change to your local data model
  • there had been a recent change on the PAT_BK column in your local data model, which you did not yet check into the repository. PowerDesigner proposes not to overwrite and revert your local change with the value from the repository

Comparing models

You can compare the model in your local workspace with latest or previous version of that model in the repository. You can also compare different versions of a model in the repository. The compare model window is similar to the Merge Changes to Repository dialog, except that you cannot update the target model with this dialog.

Customized DBMS file for HANA 2.0

For HANA 2.0 we are using a customized DBMS file in PowerDesigner. The DBMS files that are delievered with the PowerDesigner installation are located in the PD directory ~\Resource Files\DBMS. Instead, our customized SAP HANA 2.0 DB with CDWH Extensions DBMS file is embedded within the PDM_USB_HANA model and as such also saved in the PowerDesigner repository.
